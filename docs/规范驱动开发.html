SDD核心定义

规范驱动开发(Spec-driven development)意味着在编写代码之前先编写"规范"，该规范成为人类和AI的共同真相来源。

在这个新世界中，维护软件意味着演进规范...开发的通用语言提升到更高层次，代码成为最后一英里的实现方式。

关键工具对比

Kiro

轻量级VS Code扩展，遵循"需求→设计→任务"三阶段流程

Spec-kit

GitHub CLI工具，提供"宪法→规范→计划→任务"循环工作流

Tessl框架

唯一真正探索规范即源代码(Spec-as-source)方法的工具

SDD实施层级

规范优先(Spec-first)

先编写详细规范，再用于AI辅助开发

规范锚定(Spec-anchored)

任务完成后保留规范，用于功能演进

规范即源(Spec-as-source)

规范成为主要维护对象，人类不再直接编辑代码

理解规范驱动开发：Kiro、Spec-kit和Tessl的实践探索

在当前AI编程的快速发展中，"规范驱动开发"(SDD)成为最新流行术语。本文通过分析三种自称为SDD工具的产品(Kiro、spec-kit和Tessl)，试图厘清这一概念的实际含义和应用现状。

规范是什么？

规范是使用自然语言编写的结构化、面向行为的产物(或一组相关产物)，表达软件功能并作为AI编码代理的指导。目前最接近一致的定义是将规范类比为"产品需求文档"。

需要区分规范与代码库的更通用上下文文档(如规则文件或高级描述)。某些工具称后者为"记忆库"(memory bank)，这些文件与代码库中的所有AI编码会话相关，而规范仅与创建或更改特定功能的任务相关。

工具工作流分析

Kiro工作流

需求→设计→任务三阶段，每个阶段对应一个Markdown文档。包含"记忆库"概念(称为"steering")，默认包含product.md、structure.md和tech.md三个文件。

Spec-kit工作流

宪法→规范→计划→任务的循环流程。"宪法"(constitution)作为高级原则的不可变规则库。每个工作步骤生成多个Markdown文件，并使用检查清单跟踪用户澄清、宪法违规等。

Tessl框架

唯一明确追求规范锚定方法并探索规范即源代码层级的工具。规范与代码文件保持1:1映射关系，代码文件顶部标注"// GENERATED FROM SPEC - DO NOT EDIT"。

挑战与观察

  • 通用工作流程问题：现有工具提供的固定工作流难以适应不同规模问题，小型修复可能产生过多文档
  • Markdown审查负担：相比审查代码，大量生成的Markdown文件更加冗长乏味
  • 控制幻觉：尽管上下文窗口扩大，AI代理仍可能忽略部分指令或过度执行
  • 功能与技术规范分离：实践中难以明确区分功能规范和技术实现细节
  • 目标用户模糊：不清楚SDD是为开发者设计还是促进开发与产品的跨职能协作

历史经验借鉴

规范即源代码(Spec-as-source)与模型驱动开发(MDD)有重要相似之处。MDD使用自定义UML或文本DSL作为规范，通过代码生成器转换为代码，但因抽象层级尴尬和约束过多未能普及。

LLM消除了MDD的许多约束，但也引入非确定性。我们可能同时面临MDD的僵化性和LLM的不确定性双重挑战。

关键洞见

规范优先原则在AI辅助编程中具有明确价值，但"规范驱动开发"仍缺乏统一定义。现有工具可能过于字面化地复制传统工作流，反而放大了审查负担和幻觉问题。有效的SDD工具需要支持多种工作流以适应不同问题规模，并提供卓越的规范审查体验。
